# AI 产品经理作品集网站设计文档

日期：2026-06-10

## 1. 背景与定位

本项目是一个面向 AI 产品经理实习面试的个人作品集网站。它不只是展示项目链接，而是把上一段实习中的 AI 原型 Demo、PRD、项目思考和访问数据沉淀为一个可持续维护的作品集系统。

作品集的核心叙事是：候选人能够把抽象需求转化为可讨论、可验证的产品原型，并通过 PRD 和复盘表达产品判断。网站本身也会被当作一个小型产品来运营，通过访问行为数据观察不同公司、岗位和 JD 对项目内容的兴趣。

## 2. 用户与场景

### 面试官

面试官通过简历、投递系统、邮件、内推消息或专属投递链接访问公开作品集。面试官需要在较短时间内确认：

- 候选人是谁。
- 候选人与 AI 产品经理岗位的相关性。
- 候选人做过哪些真实原型和 PRD。
- 候选人是否具备产品判断、表达和复盘能力。

### 作者本人

作者需要长期维护项目内容，包括新增、删除、修改项目，上传或粘贴 Markdown PRD，配置原型链接，更新经历摘要，创建重点岗位投递链接，并查看访问行为数据。

## 3. 第一版目标

- 构建一个公开面试官页，展示个人信息、精选项目、项目证据卡片、站内 PRD 深读和经历摘要。
- 构建一个真实后台，支持白名单邮箱登录，不开放注册。
- 支持项目管理：新增、编辑、删除、排序、公开/隐藏、精选、上传截图、配置 Demo 链接、维护 PRD 和项目思考。
- 支持重点岗位投递追踪链接，例如 `/v/bytedance-ai-pm-2026`。
- 支持访问行为监控，重点观察访客对哪些项目感兴趣、对哪些项目不感兴趣，以及 PRD 阅读细节。
- 支持一键导出项目、PRD、项目思考、投递记录和统计数据备份。

## 4. 非目标

第一版不做以下内容：

- 不做多用户 SaaS，不开放注册。
- 不生成简历 PDF。简历 PDF 由作者自己维护。
- 不依赖 Vercel Analytics 作为主数据源。
- 不做 AI 自动诊断、AI 优化建议或自动判断项目策略。
- 不采集浏览器、操作系统、语言、屏幕尺寸等低价值环境字段作为核心分析指标。
- 不把原型 Demo 直接 iframe 嵌入主页面作为默认展示方式。

## 5. 公开面试官页设计

### 视觉方向

公开页采用“编辑部作品集”气质：克制、专业、有叙事感，像一份可交互的产品案例刊物。它不做强烈的 AI 炫技风，也避免模板化的紫色渐变和空泛口号。

### 首屏结构

首屏采用混合式结构：

- 左侧或主区域展示个人信息。
- 右侧或首屏下缘露出 1-2 个精选项目预览。
- 面试官进入页面后能立刻知道“这个人是谁”和“有什么真实产出可看”。

首屏默认文案：

> 如果你只有 3 分钟认识我，可以先看看这些从“能不能做个 Demo”开始的 AI 产品原型。

辅助文案：

> 这里整理了我上一段实习中的部分原型 Demo、PRD 和项目思考：它们记录了我如何把抽象需求变成可讨论的页面，也记录了我对场景、边界和协作方式的判断。

首页保留个人开场、AI 产品经理实习生候选人身份、项目证据入口和经历摘要，但不放“当前身份 / 求职方向 / 作品集内容”三块信息卡。

### 项目证据卡片

每个项目以证据卡片形式展示：

- 项目封面截图，代表原型首页预览。
- 一句话项目结论。
- 我的贡献。
- AI 使用方式。
- 关键判断或取舍。
- PRD 摘要。
- Demo 链接。
- 详情页入口。

项目卡片支持快速展开摘要。需要深读时，面试官进入项目详情页。

### 项目详情页

项目详情页展示：

- 原型截图和 Demo 链接。
- 完整 Markdown PRD，站内渲染。
- PRD 目录和章节锚点。
- 完整项目思考。
- AI 工作流说明。
- 关键判断和取舍。
- 返回首页入口。

### 简历信息

作品集不是简历替代品。网站只保留轻量经历摘要，不提供简历下载，也不完整嵌入 PDF。

经历摘要展示 3-6 条与 AI PM 最相关的信息，例如：

- 上一段实习参与团队工作流 AI 化转型。
- 主动使用 AI 搭建产品 Demo。
- 沉淀 PRD、项目复盘和产品-研发 Agent 协作探索。
- 核心能力包括场景识别、问题定义、AI 边界判断和可验证产出。

### 页脚数据提示

页脚最不显眼位置展示轻量说明：

> 我会用匿名访问数据观察这个作品集哪里更有用。如果你也好奇这些数据说明了什么，欢迎把它变成一道面试题。

## 6. 后台设计

后台是作者长期维护作品集的管理台，登录后才能访问。后台首页优先展示数据驾驶舱，而不是项目管理。

### 登录与权限

- 使用 Supabase Auth。
- 只允许作者本人白名单邮箱登录。
- 不开放注册。
- 后台页面、管理 API、备份导出和访问明细只允许管理员访问。

### 数据驾驶舱

数据驾驶舱只呈现真实行为事实，不做 AI 诊断或自动建议。

核心视图：

- 时间范围筛选。
- 投递链接筛选。
- 岗位标签筛选。
- 项目筛选。
- CSV 或 JSON 导出。

核心指标：

- 访问会话数。
- 项目展开数。
- PRD 深读数。
- Demo 点击数。
- 回访会话数。

项目兴趣明细：

- 项目曝光。
- 项目展开。
- 项目详情进入。
- PRD 深读。
- Demo 点击。
- 平均停留时间。

阅读漏斗：

- 访问首页。
- 项目曝光。
- 展开项目。
- PRD 深读。
- Demo 点击。

投递链接对比：

- 不同公司或岗位链接最常看的项目。
- 每个投递链接对应的 PRD 阅读数量。
- 每个投递链接对应的 Demo 点击数量。

PRD 阅读细节：

- 章节曝光。
- 章节阅读深度。
- 章节停留时间。
- 目录点击。

访客会话路径：

- 首页到项目卡片。
- 项目卡片到详情页。
- 详情页到 PRD 章节。
- PRD 到 Demo。

低兴趣内容：

- 曝光高但展开低的项目。
- 展开后停留短的项目。
- 到达率低的 PRD 章节。
- Demo 点击低的项目。

展示模式：

- 展示项目兴趣、投递链接对比、阅读漏斗和 PRD 阅读细节。
- 隐藏原始 IP 明细。
- 可保留省市和可能来源作为访问背景。

### 项目 / PRD 管理

后台支持项目增删改查和排序。

项目字段：

- 标题。
- slug。
- 标签。
- 一句话项目结论。
- 原型链接。
- 封面截图。
- 我的贡献。
- AI 使用方式。
- 关键判断或取舍。
- 项目思考。
- Markdown PRD 原文。
- PRD 渲染预览。
- 公开状态。
- 精选状态。
- 首页排序。
- 是否参与统计。

PRD 支持：

- 粘贴 Markdown。
- 上传 Markdown 文件。
- 站内渲染预览。
- 自动生成目录和章节锚点。
- 在详情页记录章节阅读事件。

原型预览采用截图式展示。后台支持手动上传截图，后续可以扩展自动抓取截图。

### 投递追踪链接管理

作者可以在后台创建投递记录。

投递记录字段：

- 公司名称。
- 岗位名称。
- JD 链接或 JD 摘要。
- 岗位标签。
- 投递渠道。
- 备注。
- campaign slug。
- 是否启用。
- 创建时间。
- 更新时间。

生成链接格式：

部署域名由上线时决定，路由结构固定为 `/v/[campaignSlug]`。例如部署到 `https://portfolio.vercel.app` 时，重点岗位链接可以是 `https://portfolio.vercel.app/v/bytedance-ai-pm-2026`。

访问专属链接后，面试官看到的是同一个公开作品集页。系统记录 campaign 归因，并把后续访问事件关联到该投递记录。

第一版支持投递链接统计，不做复杂自动个性化。可以保留字段用于未来配置特定投递链接的精选项目顺序。

### 经历摘要管理

后台支持维护公开页的个人信息和经历摘要：

- 姓名。
- 候选人身份。
- 联系方式。
- 首屏主文案。
- 首屏辅助文案。
- 经历摘要条目。

### 备份导出

后台支持导出备份：

- 项目配置。
- Markdown PRD 原文。
- 项目思考。
- 投递记录。
- 访问统计数据。
- 经历摘要。

第一版导出 `.json`。截图文件打包为 `.zip` 可作为后续增强。

## 7. 数据采集设计

### 采集事件

第一版采集以下事件：

- 页面访问。
- 会话开始和会话结束。
- 回访识别。
- 项目曝光。
- 项目曝光时长。
- 项目展开。
- 项目详情进入。
- PRD 摘要展开。
- PRD 全文进入。
- PRD 章节曝光。
- PRD 章节停留时间。
- PRD 滚动深度。
- Demo 点击。
- 外链点击。
- 经历摘要曝光和停留。

不采集作为核心分析的字段：

- 浏览器。
- 操作系统。
- 页面语言。
- 屏幕尺寸。

### 访客识别

系统使用匿名 visitor id 和 session id 识别回访，不要求面试官登录。

可记录的访问背景：

- campaign slug。
- referrer。
- IP。
- 省市。
- 可能来源或网络归属。

公司识别优先级：

1. 专属投递链接。
2. referrer 或 UTM。
3. IP 省市、ASN、企业网络辅助判断。

不把 IP 反推公司作为主要依据，因为邮箱、招聘平台、VPN 和移动网络都会干扰识别。

### 上报方式

前端事件先做轻量聚合，再批量写入 `/api/events`，避免每个小动作都直接打数据库。

高频行为如悬停或焦点停留不逐条记录。第一版只记录对作品集优化有明确价值的阅读和点击行为。

## 8. 技术架构

### 技术栈

- Next.js App Router。
- TypeScript。
- Supabase Auth。
- Supabase Postgres。
- Supabase Storage。
- Vercel 部署。

### 路由

- `/`：公开面试官页。
- `/projects/[slug]`：项目详情页。
- `/v/[campaignSlug]`：重点岗位专属追踪链接。

- `/admin/projects`：项目和 PRD 管理。

- `/admin/profile`：个人信息与经历摘要。
- `/admin/backups`：备份导出。
- `/api/events`：行为事件上报。

### 安全策略

- Supabase RLS 默认开启。
- 公开访问只能读取已发布项目、公开经历摘要和必要展示内容。
- 后台写入和读取管理数据仅允许管理员。
- 访问事件上报 API 只允许写入受控事件字段。
- 不在前端暴露 Supabase service role key。
- Storage 中公开项目截图可公开读取，后台上传需要管理员权限。

## 9. Vercel 与 Supabase 使用边界

Vercel 负责托管 Next.js 应用、路由和 API。专属追踪链接不是 Vercel 自动生成，而是后台在 Supabase 中创建 campaign 记录后展示出来的应用路由。

Supabase 作为主数据源，存储项目、PRD、投递记录、访问事件和备份数据。

Vercel 免费版可用于启动第一版。Supabase Free Plan 长时间低活跃可能暂停项目，因此正式投递期建议使用 Supabase Pro，或至少保持后台备份导出能力。

## 10. 验收标准

公开页：

- 面试官无需登录即可访问公开页。
- 公开页能展示个人信息、首屏文案、精选项目、项目证据卡片、经历摘要和页脚数据提示。
- 项目卡片能展开摘要，并能进入项目详情页。
- 项目详情页能渲染 Markdown PRD 和项目思考。
- Demo 链接能打开真实 GitHub Pages 或 Vercel 原型。

后台：

- 只有白名单邮箱能登录后台。
- 作者能新增、编辑、隐藏、删除和排序项目。
- 作者能粘贴或上传 Markdown PRD，并预览渲染结果。
- 作者能上传或配置项目封面截图。
- 作者能创建投递追踪链接。
- 作者能查看项目兴趣、投递链接对比、阅读漏斗、PRD 阅读细节和访客会话路径。
- 作者能导出备份数据。

数据采集：

- 访问 `/v/[campaignSlug]` 后，事件能关联到对应投递记录。
- 项目曝光、展开、详情进入、PRD 阅读和 Demo 点击能被记录。
- 展示模式隐藏原始 IP 明细。

安全：

- 未登录用户不能访问后台页面。
- 未登录用户不能读取管理数据。
- 公开页不能展示草稿或隐藏项目。
- 前端不暴露服务端密钥。
